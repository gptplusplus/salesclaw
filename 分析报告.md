**风险点 1：时间窗口匹配**

**证据**：定义了 `xxx` 事件表，包含 `生效日 × 失效日 × 变更原因`，但未明确索引设计。

**风险点 2：流向归因**

流向数据聚合医院维度时，需要正确处理"同一品种×同一医院"的流向拆分。若存在多代表分管同一医院的情况，需要按 `xxx_id` 拆分的费用数据与流向数据进行关联，容易产生重复计算或遗漏。

**证据**：`人` 与 `医院` 通过"分管事实"连接，但"分管"边界未在 schema 层面约束。

**风险点 3：跨库状态传播**

VBP 分析需要从政策事实一直传播到预警事件，中间经过 5 个库的派生计算。在分布式环境下，任何一个环节的数据延迟都会导致端到端状态不一致。

### 2.4 可行性评级：**条件可行**

**充分条件**：

- 路径固定，不存在分支选择，Agent 可确定性执行
- 每步操作均为标准的数据库连接/聚合操作，无复杂递归
- 业务语义清晰，支持链路追踪

**必要条件**：

- 流向数据的时间粒度（月度/季度）需与政策事件粒度对齐
- 跨库事务需支持最终一致性（允许 24h 内的数据延迟）

### 7.2 Tool-Augmented LLM 调用价值

**价值 1：来源可信度分级**

LLM 在生成答案时，可根据 `confidence` 字段调整引用权重：

```python
def generate_answer_with_provenance(query: str, field: str) -> str:
    provenance = sources[field_provenance]
    
    # 高置信度来源直接引用
    high_conf = [v for v in provenance.values if v.confidence == "high"]
    medium_conf = [v for v in provenance.values if v.confidence == "medium"]
    
    if high_conf:
        answer = f"根据{high_conf[0].source}，{field}为{high_conf[0].value}"
        if medium_conf:
            answer += f"（另有一来源{medium_conf[0].source}标注为{medium_conf[0].value}，置信度中等，建议人工确认）"
    else:
        answer = f"⚠️ 警告：{field}缺乏高置信度来源，请人工核实"
        
    return answer
```

**价值 2：答案可审计性**

当用户质疑"为什么这个字段是这个值"时，Agent 可回溯至 `source` 字段提供解释：

```python
def explain_field_value(field: str, value: Any) -> str:
    provenance = sources[field_provenance][field]
    explanation = f"字段{field}的值为{value}，来源：\n"
    for v in provenance.values:
        explanation += f"- {v.source}（置信度：{v.confidence}）\n"
        if v.note:
            explanation += f"  备注：{v.note}\n"
    return explanation
```

**价值 3：冲突检测与消解**

当多个来源给出不同值时，Agent 可自动标记冲突：

```python
def detect_conflicts(field_provenance: dict) -> List[Conflict]:
    conflicts = []
    for field, prov_data in field_provenance.items():
        values = [v.value for v in prov_data.values]
        if len(set(values)) > 1:
            conflicts.append(Conflict(
                field=field,
                values=prov_data.values,
                canonical=prov_data.canonical,
                resolution_needed=True
            ))
    return conflicts
```

2. **来源版本缺失**：仅记录文档名称，未标注具体章节版本号

## 八、算法设计亮点与不足

### 8.1 亮点

**亮点 1：静态/动态分层设计（★★★★★）**

静态层承载长期不变的属性，动态层由外部派生，这一设计从架构层面避免了"同一字段双写"的问题。Agent 推理时可根据 `layer` 字段判断数据来源，显著降低推理复杂度。

**亮点 2：AI Grade 分级（★★★★★）**

`xxx.json` 中的 `xxx`  为 Agent 提供了直接的置信度分级，直接影响推理策略选择。这一设计在企业级知识库中属于领先实践。

**亮点 3：xxx.json 字段级血缘（★★★★☆）**

`xxx` 提供了字段级别的来源追踪能力，支持冲突检测、答案审计和置信度加权，是 Tool-Augmented LLM 的理想基础设施。

**亮点 4：xxx.yaml 回归测试（★★★★☆）**

17 项 KPI 断言覆盖了品种静态层的核心字段，为数据质量提供了自动化保障机制。

### 8.2 不足

**不足 1：跨库主键对齐机制缺失（P0）**

xxx 和 xxx 在跨库关联时缺乏统一的对照表或别名机制，是当前最严重的算法障碍。

**不足 2：动态层派生缺乏版本锁定（P1）**

动态层字段依赖外部派生，但无版本锁定机制。数据回溯修正后，动态层不会自动重新计算，导致端到端状态不一致。

**不足 3：元数据自动化程度低（P1）**

xxx、xxx.json、xxx 均为人工维护或半自动生成，缺乏端到端的自动化血缘追踪管道。

**不足 4：逆边缺失导致推理不对称（P2）**

图谱中仅有xxx 的正向索引边，缺少逆边，Agent 无法从人员/医院反向查询品种列表。

**不足 5：时序数据建模不足（P2）**

政策事件、流向数据均有时序特征，但图谱关系边未标注时间维度，Agent 无法进行"政策调整前后的流向对比"等时序推理。
