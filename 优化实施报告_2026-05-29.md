# SalesClaw Ontology 优化实施报告

> 日期：2026-05-29

---

## 一、优化背景

数据结构已搭建，但 Palantir 式的语义闭环远未打通。

### 核心断点

```
现场问题 → 业务对象 → 对象关系 → [断点1]业务动作 → [断点2]权限治理 → 可运行应用 → [断点3]AI执行 → 反馈迭代
```

- **断点1**：动作不执行（链路在"业务动作"处断裂）
- **断点2**：没有权限治理（链路在"权限治理"处断裂）
- **断点3**：AI 不在 Ontology 上工作（链路在"AI执行"处断裂）

## 三、六大优化方向详解

### 优化点 1：对象获得"业务身份"——负责人与生命周期


#### 1.1 OntologyObject 增加 owner_id 和 stakeholders

**文件**：`backend/models/ontology.py`

**影响范围**：
- `backend/schemas/ontology.py`：Create/Update/Response 三个 Schema 均增加 `ownerId` 和 `stakeholders`
- `backend/services/ontology_service.py`：create/update/response 三个路径均处理新字段
- `backend/seed.py`：每个种子对象填充 owner_id（医生/医院→r1，其余→default_user）
- `backend/database.py`：自动迁移，添加列和索引

#### 1.2 生命周期状态机

**新文件**：`backend/services/lifecycle_service.py`

定义了 7 种对象类型的合法状态转换路径：

| 对象类型 | 合法路径 |
|---------|---------|
| Doctor | prospect → developing → mature → at_risk → recovering → mature；at_risk → churned |
| Hospital | prospect → contact → negotiation → contract → launch → mature → at_risk → recovering → mature；at_risk → churned |
| SalesRep | new → active → at_risk → recovering → active；at_risk → churned |
| Product | launched → growing → mature → declining → discontinued |
| RWSProject | initiated → multicenter → gcp → settlement → completed |
| PDCAPlan | planning → doing → checking → acting → completed |
| CustomerCategory | new → active → at_risk → recovering → active；at_risk → churned |

#### 1.3 后端 Ontology 定义加载器

**新文件**：`backend/services/ontology_definition.py`

将前端 `DomainOntology.ts` 的定义镜像到后端，包含 5 个域的完整定义：

- **RevenueManagement**：6 个概念、7 个关系、5 条公理
- **CustomerManagement**：6 个概念、6 个关系、4 条公理
- **ExpenseManagement**：5 个概念、5 个关系、3 条公理
- **MedicalAffairs**：4 个概念、4 个关系、3 条公理
- **ComplianceManagement**：4 个概念、4 个关系、4 条公理

### 优化点 2：关系从"哑线"变为"活线"

**新文件**：`backend/services/link_service.py`

#### 2.1 反向关系自动创建

定义了 **40 对**双向关系映射，核心示例：

| 正向关系 | 反向关系 | 语义 |
|---------|---------|------|
| WORKS_AT | CONTAINS_MEMBER | 医生→医院 / 医院→医生 |
| PRESCRIBES | PRESCRIBED_BY | 医生→产品 / 产品→医生 |
| MANAGED_BY | MANAGES | 对象→代表 / 代表→对象 |
| INFLUENCES | INFLUENCED_BY | KOL→对象 / 对象→KOL |
| COMPLIES_WITH | GOVERNS | 对象→规则 / 规则→对象 |
| VIOLATES | VIOLATED_BY | 对象→规则 / 规则→对象 |

**集成点**：
- `ontology_service.create_object`：创建 Link 后自动调用 `create_inverse_link`
- `ontology_service.delete_object`：删除 Link 时同步删除反向 Link
- `inference_service._apply_conclusion`：推理创建 Link 后自动创建反向

#### 2.2 基数校验

定义了 18 种关系的基数约束：

| 基数 | 示例关系 | 含义 |
|------|---------|------|
| 1:1 | STRATEGY_FOR | 一个对象只能有一个策略 |
| 1:N | MANAGES, CONTAINS | 一个代表管理多个医生 |
| N:1 | WORKS_AT, BELONGS_TO | 多个医生属于一个医院 |
| N:M | PRESCRIBES, INFLUENCES | 多对多 |

**集成点**：`ontology_service.create_object` 创建 Link 前校验基数，违反则跳过

#### 2.3 有效期过滤

- `filter_expired_links(links)` → 过滤 `valid_to < now()` 的过期关系
- **集成点**：`ontology_service._batch_load_related_data` 加载关系后自动过滤

#### 2.4 推理去重与冲突检测

- `check_link_conflict(db, source_id, link_type, target_id, confidence)` → 检测重复和矛盾关系
- 矛盾对：`COMPLIES_WITH` vs `VIOLATES`、`INFLUENCES` vs `INFLUENCED_BY`（方向矛盾）
- **集成点**：`inference_service._apply_conclusion` 创建推理 Link 前检测冲突，有冲突则跳过

### 优化点 3：动作从"空壳动词"变为"可执行操作"

**新文件**：`backend/services/action_executor.py`

#### 3.1 Action 执行引擎架构

```
execute_object_action(object_id, action_name, params, user_id)
  ├── 1. 查找对象和动作定义
  ├── 2. 解析并校验前置条件 (preconditions)
  ├── 3. 执行核心动作逻辑 (15 种 Action Handler)
  ├── 4. 执行副作用链 (side_effects)
  ├── 5. Write-Back 到外部系统
  ├── 6. 记录事件 (ObjectEvent)
  ├── 7. 记录审计日志 (AuditLog)
  └── 8. 记录执行日志 (ExecutionLog)
```

#### 3.2 15 种 Action 实现

| Action | 对象类型 | 功能 | 副作用 |
|--------|---------|------|--------|
| scheduleVisit | Doctor | 安排拜访 | 更新 nextRecommendedVisitDate |
| updateSentiment | Doctor | 更新态度 | 更新 sentiment 字段 |
| flagComplianceRisk | Doctor | 标记合规风险 | 创建 ComplianceAlert |
| markAsAtRisk | Doctor/Hospital | 标记流失风险 | 更新 lifecycleStage + status |
| generateVisitBrief | Doctor | 生成拜访简报 | 创建 VisitBrief |
| updateAccessStatus | Hospital | 更新准入状态 | 更新 access_status |
| dismiss | ComplianceAlert | 忽略警报 | 更新 alert_status |
| escalate | ComplianceAlert | 升级警报 | 更新 alert_status |
| approve | RecoveryPlan | 批准计划 | 更新 plan_status |
| reject | RecoveryPlan | 驳回计划 | 更新 plan_status |
| updateActualValue | SalesTarget | 更新实际值 | 计算达成率 |
| allocateBudget | BudgetCategory | 分配预算 | 计算执行率 |
| advanceCycle | PDCAPlan | 推进周期 | 按状态机推进 |
| updateStatus | 通用 | 更新状态 | 校验生命周期转换 |
| inviteToEvent | Doctor | 邀请参会 | 发送邀请通知 |

#### 3.3 前置条件解析

支持 5 种条件运算符：

| 运算符 | 示例 | 含义 |
|--------|------|------|
| `!=` | `lifecycleStage != 'churned'` | 不等于 |
| `==` | `status == 'warning'` | 等于 |
| `>=` | `achievementRate >= 80` | 大于等于 |
| `>` | `riskScore > 0.7` | 大于 |
| `in` | `lifecycleStage in [mature, developing]` | 属于集合 |

#### 3.4 副作用链

支持 4 种副作用指令：

| 指令 | 示例 | 效果 |
|------|------|------|
| `create` | `create VisitRecord` | 创建新对象 |
| `update` | `update doctor.nextRecommendedVisitDate` | 更新字段 |
| `notify` | `notify managedBy` | 发送通知 |
| `trigger` | `trigger inference_engine` | 触发下游引擎 |

#### 3.5 Write-Back 机制

- 通过环境变量 `WEBHOOK_{TARGET_SYSTEM}` 配置外部系统 Webhook
- 当前记录 Write-Back 意图到 ExecutionLog，后续可对接真实系统
- 支持的目标：CRM、ERP、ComplianceSystem

#### 3.6 集成点

- `routers/ontology.py` 的 `execute_action` 端点：从手动日志记录改为调用 `ActionExecutor`
- `services/action_service.py` 的 `execute_action`：从 3 种硬编码 Handler 改为统一 `ActionExecutor`

### 优化点 5：AI 站在 Ontology 上工作——OAG

**新文件**：`backend/services/ontology_context.py` + `backend/services/oag_service.py`

#### 5.1 Ontology Context Provider

给定一个对象 ID，自动展开其完整业务上下文：

```
build_ontology_context(object_id, depth=2)
  ├── 对象属性 + 状态 + 生命周期 + 负责人
  ├── 一度关系（直接 Link）+ 二度关系（Link 的 Link）
  ├── 可执行动作 + 前置条件 + 参数
  ├── 近期事件（最近 10 条）
  ├── 时序数据趋势（变化方向 + 百分比）
  └── Ontology 定义（概念描述 + 接口 + 属性）
```

`format_context_for_llm()` 将结构化上下文格式化为中文 Markdown，适合注入 LLM System Prompt。

#### 5.2 OAG（Ontology-Augmented Generation）

区别于传统 RAG（检索文档），OAG 检索的是**业务对象和关系**：

```
用户提问 → 查找相关对象 → 展开 Ontology 上下文 → 注入 LLM Prompt → 生成基于事实的回答
```

**OAG System Prompt 核心指令**：
1. 引用具体对象和关系（如"张主任 → WORKS_AT → 瑞金医院"）
2. 基于时序数据趋势做判断
3. 如有风险，指出风险传播路径
4. 给出可执行的建议，映射到 Ontology 中的 Action
5. 明确哪些操作需要审批

#### 5.3 对象深度分析

`oag_object_analysis(object_id)` 产出三维度分析：

| 维度 | 检测项 |
|------|--------|
| 风险 | 状态风险、生命周期风险、合规风险、态度风险、趋势风险 |
| 机会 | 增长机会、趋势机会、KOL 影响力机会 |
| 动作建议 | 基于风险等级推荐 Action，含优先级和审批要求 |

#### 5.4 Agent 与 Ontology 对齐

**文件**：`backend/agents/cognitive_graph.py`

| Agent 节点 | 优化前 | 优化后 |
|-----------|--------|--------|
| perceive | 只扫描异常数量 | 注入 Ontology 上下文，理解关系链 |
| reason | 关键词匹配推理类型 | 基于 Ontology 上下文推理 |
| plan | 自由文本计划 | 计划映射到 Ontology Action（格式：`ACTION_NAME on OBJECT_ID with PARAMETERS`） |
| execute | 直接操作数据库 | 通过 ActionExecutor 执行，走 Ontology Action 管线 |

#### 5.5 新增 API 端点

| 端点 | 功能 |
|------|------|
| `POST /api/oag` | OAG 对话 |
| `GET /api/oag/analyze/{object_id}` | 对象深度分析 |

### 优化点 6：系统从"工具"进化为"运营层"

#### 6.1 Ontology 变更管理

**新文件**：`backend/services/ontology_change_service.py`

| 功能 | 说明 |
|------|------|
| 提交变更请求 | 自动分析影响范围（受影响关系数、风险等级） |
| 审批变更 | approve/reject，记录审批人和时间 |
| 查询待审批 | 列出所有 pending 状态的请求 |

影响分析逻辑：
- 受影响关系 > 10 → 高风险
- 受影响关系 > 3 → 中风险
- 删除操作自动提升风险等级

#### 6.2 Ontology 版本管理

| 功能 | 说明 |
|------|------|
| 创建版本快照 | 捕获所有对象、关系、动作的完整状态 |
| 版本历史 | 列出最近 20 个版本 |
| 版本对比 | 计算两个版本间的对象/关系增删 |

#### 6.3 Ontology 健康度监控

**新文件**：`backend/services/ontology_health.py`

健康度评分算法（满分 100）：

| 问题类型 | 单项扣分 | 上限 |
|---------|---------|------|
| 孤立对象（无 Link） | 5 分/个 | 30 分 |
| 僵尸动作（定义未执行） | 3 分/个 | 20 分 |
| 缺失属性（必填为空） | 4 分/个 | 25 分 |
| 悬空链接（目标不存在） | 8 分/个 | 25 分 |

自动生成优先级建议：
- **高**：补充缺失属性、清理悬空链接
- **中**：为孤立对象添加关系
- **低**：审查僵尸动作

| 事件类型 | 触发时机 |

| object.created | 创建对象 |

#### 7.2 内存缓存层

**新文件**：`backend/services/cache_service.py`

| 特性 | 说明 |
|------|------|
| 默认 TTL | 300 秒（5 分钟） |
| 最大容量 | 500 条 |
| 淘汰策略 | LRU（按创建时间淘汰最旧） |

缓存键设计：
- `obj:{object_id}` → 对象数据（TTL 300s）

**集成点**：`ontology_service.get_object_by_id` 先查缓存，命中则跳过 DB 查询；create/update/delete 时自动失效
